## Corpus A — Small Files (100.000 archivos, seed 42)

- 100.000 archivos, 274 MB, **tamaño medio 2.740 bytes**, 15.239 duplicados,
  6.418 carpetas, 111.060 operaciones ejecutables.

El pipeline completo tarda ~653 s; **la ejecución es el 80,7 %**. Todo lo
demás (escaneo, hash, análisis, verificación) suma menos de una quinta parte.

### Ejecución: dónde se va el tiempo (medido, no supuesto)

Desglose de `stage_nanos` sobre las 111.060 operaciones del protocolo §27.1:

| Etapa | Segundos | % pared | µs/op | Categoría |
| --- | ---: | ---: | ---: | --- |
| persist_result | 63,9 | 12,1 % | 575 | **SQLite** |
| sync_all | 58,9 | 11,2 % | 531 | durabilidad (fsync) |
| create_partial | 58,5 | 11,1 % | 527 | filesystem |
| lease | 58,1 | 11,0 % | 524 | **SQLite** |
| finalize | 52,9 | 10,0 % | 476 | filesystem (rename) |
| claim_persist | 48,2 | 9,1 % | 434 | **SQLite** |
| resolve_destination | 31,3 | 5,9 % | 282 | rutas + frontera segura |
| **copy_stream** | **30,0** | **5,7 %** | 270 | **copia real de bytes** |
| revalidate_root | 14,8 | 2,8 % | 133 | frontera segura |
| post_fingerprint | 3,9 | 0,7 % | 35 | huella origen §14.5 |
| preflight_source | 2,5 | 0,5 % | 23 | huella + parse manifiesto |
| collision_check | 1,0 | 0,2 % | 9 | §27.3 |
| *(no atribuido)* | ~103 | 19,6 % | — | batch queries, reclaim, mark_running, muestreo |

### Conclusión del cuello de botella

Agrupando por naturaleza del coste:

- **SQLite por operación (lease + claim + result): ~170 s, 32 %.** Tres
  transacciones separadas por archivo. Es la mayor categoría única.
- **Latencia de syscalls por archivo (create_partial + sync_all + finalize):
  ~170 s, 32 %.** Dominadas por *latencia por llamada*, no por ancho de banda.
- **Copia real de bytes (copy_stream): 30 s, apenas 5,7 %.** Con 2,7 KB de
  media, mover los datos es lo de menos.
- **Frontera segura (resolución + revalidación + huellas): ~54 s, 10 %.**

Esto **confirma la hipótesis del encargo**: el límite no es el ancho de banda
del SSD (0,5 MiB/s en ejecución con un NVMe que hace GB/s), sino la latencia
por archivo y los commits individuales. El corpus de 1M de M0.8 fue aún más
lento por operación (54 vs 189 archivos/s); parte de esa diferencia es el
crecimiento de la base SQLite, lo que **agrava** la categoría SQLite a escala.

### Implicaciones para las siguientes fases

1. **Reutilizar buffers** (fase 2) moverá poco el corpus A —`copy_stream` es
   solo 5,7 %— pero es correcto y ayudará al corpus C (archivos grandes).
2. Las ganancias grandes están en (a) **agrupar los commits SQLite** en
   microlotes compatibles con la recuperación y (b) **paralelizar** las
   etapas de filesystem independientes por archivo (create/sync/finalize),
   que es la propuesta de **ejecución estricta paralela** — a construir
   desde el modelo de recuperación, no envolviendo el bucle en Rayon.
3. `sync_all` (durabilidad) no se elimina en modo estricto; se puede
   paralelizar y, en un futuro perfil `fast` explícito, diferir.

## Corpus D — Million Entries

Referencia existente de M0.8 (`docs/testing/m0.8-scale-1m.md`): 1.000.000
